# Helix End-to-End Framework Guide

This guide shows the practical, repo-native workflow for shipping a new feature/page in Helix from data model to UI interactions.

## What this guide covers

- Server data (`resource`) and mutations (`action`)
- TSX shell view + compiled artifact generation
- Server wrappers, handlers, and route wiring
- Client bindings and optional hybrid JSON updates
- Validation, drift checks, and troubleshooting

## Prerequisites

- Node + npm installed
- Dependencies installed: `npm install`
- Build succeeds locally: `npm run build`

## Recommended pattern (default)

Use this split for new pages:

1. TSX shell in `src/example/views-tsx/*.view.tsx`
2. Generated compiled artifact in `src/example/views/compiled/*.compiled.ts`
3. Server wrapper in `src/example/views/*.ts`
4. Optional client module + JSON endpoint for high-frequency updates

Reference page: `/host-listings`

## End-to-end implementation steps

## 1) Define query + data shape

If your page supports paging/sorting/filtering, add query parsing helpers in:

- `src/example/utils/query.ts`

Define a typed context for the resource key and fetch logic.

## 2) Add/update a resource

Create or extend a resource in:

- `src/example/resources/*.resource.ts`

Checklist:

- Stable key function
- Correct cache + dedupe policy
- `tags` for invalidation
- Typed page payload (`rows`, paging metadata, filters/sort metadata)

## 3) Add/update actions for writes (if needed)

Define action(s) in:

- `src/example/resources/actions.ts` (or relevant module)

Checklist:

- Declare `invalidates`
- Declare capabilities
- Use idempotency where needed
- Keep server as durable source of truth

## 4) Author TSX shell view

Create:

- `src/example/views-tsx/your-page.view.tsx`

Guidelines:

- Keep shell structure static
- Use `<hx-slot value={...} />` for dynamic HTML sections (table/cards/pager)
- Keep stable `data-hx-id` anchors for targets that client patching may update

## 5) Generate compiled views

Run:

```bash
npm run gen:views
```

This emits:

- `src/example/views/compiled/your-page.compiled.ts`

Drift check:

```bash
npm run check:views
```

## 6) Add server wrapper view

Create/extend:

- `src/example/views/your-page.ts`

Pattern:

- Build dynamic HTML (rows/cards/pager)
- Call `renderYourPageCompiledView({ ...props })`

## 7) Add page handler and route

Update:

- `src/example/handlers/pages.handler.ts`
- `src/example/routes/index.ts`

Checklist:

- Parse URL/query
- Read resource
- Render wrapper view
- Return SSR response

## 8) Add optional hybrid JSON update path

For high-frequency interactions (paging/sort/filter), add:

- JSON endpoint in `src/example/handlers/api.handler.ts`
- Route entry in `src/example/routes/index.ts`
- Client module in `src/example/client/*.ts`
- Bootstrap import in `src/example/client/bootstrap.ts`

Client behavior checklist:

- Intercept control click/input/change
- Fetch JSON (`/api/...`)
- Patch only stable DOM targets (`data-hx-id`)
- Sync URL via `history.pushState`
- Handle `popstate`
- Fallback to `window.location.assign(...)` on failures

## 9) Add binding handlers when using `data-hx-bind`

If you use `data-hx-bind`, export handlers in:

- `src/example/client/actions.ts`

Then regenerate/check bindings:

```bash
npm run gen:bindings
npm run check:bindings
```

## 10) Validate before push

Run:

```bash
npm run build
npm run test
npm run verify
```

`npm run verify` is the final gate (imports, boundaries, generated drift, tests).

## Generic code in `src/example` that can move into `src/helix`

Use this as a promotion checklist: if logic is reusable across multiple pages/apps, move it to `src/helix`.

| Current location | Generic opportunity | Candidate framework destination |
| --- | --- | --- |
| `src/example/client/external-grid.ts` | Reusable client controller for JSON paging/sorting/filtering + history sync | `src/helix/client-router.ts` (or new `src/helix/list-controller.ts`) |
| `src/example/utils/query.ts` (`parsePage`, sort/filter parsing patterns) | Shared query parser helpers and typed page/filter state builders | new `src/helix/query.ts` |
| `src/example/utils/http.ts` (form error shaping pattern) | Generic server helper for building `FormActionError` responses | `src/helix/form.ts` or `src/helix/node-http.ts` |
| `src/example/handlers/pages.handler.ts` (`/components/app-core` fragment flow) | Generic SSR fragment endpoint helper for app-shell navigation | `src/helix/ssr.ts` or new `src/helix/server-fragments.ts` |
| `src/example/handlers/static.handler.ts` (mount list pattern) | Factory for static mounts so apps only declare prefixes/dirs | `src/helix/node-http.ts` |

### Example promotion candidate (concrete)

The Host Listings flow demonstrates the kind of logic that should become framework-level when repeated:

1. Parse query state (`page`, `sort`, `filters`)
2. Read server resource with that state
3. Return SSR HTML on first load
4. Return JSON for incremental updates
5. Patch stable `data-hx-id` targets client-side
6. Sync history with `pushState`/`popstate`

If this exact pattern appears on 2+ pages, extract it into a generic Helix utility instead of duplicating page-level client modules.

### Promote vs keep in example

- Promote to `src/helix` when code is page-agnostic and reusable across apps.
- Keep in `src/example` when logic is domain-specific (labels, columns, API shape, business rules).

## Troubleshooting matrix

| Symptom                                      | Likely cause                                             | Fix                                                                         |
| -------------------------------------------- | -------------------------------------------------------- | --------------------------------------------------------------------------- |
| `Compiled view drift detected ...`           | TSX source changed but compiled artifact not regenerated | Run `npm run gen:views`, then `npm run check:views`                         |
| `Binding map drift detected ...`             | `data-hx-bind` changed or handler exports changed        | Run `npm run gen:bindings`, then `npm run check:bindings`                   |
| Click handler never fires                    | `data-hx-bind` id doesn’t match exported handler mapping | Verify bind id in HTML and export name in `actions.ts`; regenerate bindings |
| Pager/filter URL updates but content doesn’t | Missing client module import or wrong target selector    | Check `bootstrap.ts` import and `data-hx-id` selectors                      |
| Back/forward breaks on hybrid page           | `popstate` handler missing/conflicting                   | Add page-specific `popstate` handling and guard app-level handler if needed |
| Full-page reload on interaction              | Fetch/patch path threw and fallback navigation executed  | Inspect network/console; fix API route or DOM target mismatch               |
| Verify fails on where-boundaries             | Client imported server-only resource/action              | Move import to server code path and call via API/action boundary            |

## Design guardrails

- Keep durable state and validation server-side.
- Keep client state to transient interaction concerns.
- Prefer stable `data-hx-id` targets over broad DOM replacement.
- Use generated artifacts (`gen:views`, `gen:bindings`) as source-of-truth outputs.
- Treat `npm run verify` as the required pre-push check.
